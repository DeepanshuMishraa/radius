## Problem Statement

Existing email clients are cluttered, slow, and visually overwhelming. Gmail's web UI is the worst offender — promos, social tabs, chat widgets, meet integrations, and ads all compete for attention. Users who want to simply *read their email* are forced through a gauntlet of distractions.

There is no email client that feels genuinely calming to use — where performance and visual simplicity are the primary features, not afterthoughts.

## What Makes This Cool

Radius is built on Electrobun (Bun runtime + system WebView) — not Electron. Bun's native SQLite driver (`bun:sqlite`) eliminates the dependency chain that bloats every other desktop email client. The sync engine runs in Electrobun's main process as plain TypeScript. The UI is React + Tailwind in the renderer, communicating over IPC.

The result: a desktop email client where the entire backend — auth, sync, caching — is a few hundred lines of TypeScript with zero native compilation steps. No Rust toolchain. No Node.js dependency tree. Just Bun, SQLite, and the Gmail API.

The "whoa" moments:
- **Rich onboarding:** Single screen. Connect Gmail. Watch your inbox sync with a live progress bar. Then it just *appears* — and opens instantly every time after because the data is local.
- **Typographic reader mode:** The email body rendered in a centered 680px column, system serif, generous line height. Most clients treat the message view as an afterthought. Radius makes it the hero.
- **Zero-lag feel:** The Gmail API is slow. The user never knows. The cache is always warm. Scrolling through 10k emails at 120fps via virtualization.

## Constraints

- Electrobun (Bun main process + system WebView renderer)
- React 18 + Tailwind CSS 4 + Vite
- Gmail-only for v0.1 (OAuth, read-only inbox + thread view)
- No compose, no send, no archive, no labels, no delete — read-only
- No AI features
- Open source (MIT)
- Must feel instant despite slow Gmail API — local SQLite cache is mandatory

## Premises

1. **Speed is THE feature.** Users switch because Radius feels instant, not because it has more features. Local SQLite cache hides Gmail API latency. Background sync means the inbox is always ready.
2. **Gmail-only, read-only is the right v0.1 scope.** OAuth → inbox list → thread view. No compose, no send, no multi-account. Read-only is focus, not a limitation.
3. **"Feel at home" means subtraction, not addition.** Opinionated defaults. No 40-toggle settings panel. No customizable sidebar. If a design decision requires user choice, the default is probably wrong.
4. **Electrobun's web stack (React + Tailwind) is fast enough with discipline.** No heavy component libraries. Virtualized lists. The bottleneck is the Gmail API, not rendering — solve that with the cache.
5. **Open source from day one.** MIT license. GitHub repo. Ship it, share it.

- **Effort:** M (3-5 days)

- **Effort:** M (3-5 days)
- **Risk:** Medium
- **Pros:** Reusable by other devs. Clean API boundary.
- **Cons:** Delayed UI. Library API design overhead for a weekend project.

## Recommended Approach

**Approach B: The Daily Driver**

The architecture splits cleanly across Electrobun's process model:

```
radius/
├── src/
│   ├── bun/                          # Main process (Bun)
│   │   ├── index.ts                  # App entry, window creation
│   │   ├── auth.ts                   # Gmail OAuth2 PKCE flow
│   │   ├── sync.ts                   # Sync orchestrator (full + incremental)
│   │   ├── gmail.ts                  # Gmail API client (list, get, history)
│   │   ├── db.ts                     # SQLite schema + queries (bun:sqlite)
│   │   └── ipc-handlers.ts           # IPC bridge: main → renderer
│   └── mainview/                     # Renderer process (React + WebView)
│       ├── App.tsx                   # Top-level routing
│       ├── components/
│       │   ├── Onboarding.tsx        # Single-screen OAuth flow
│       │   ├── SyncProgress.tsx      # First-sync progress bar
│       │   ├── InboxList.tsx         # Virtualized email list
│       │   ├── EmailRow.tsx          # Single-line: sender + subject + date
│       │   ├── ReaderView.tsx        # Typographic email reader
│       │   └── NewMailIndicator.tsx  # Subtle notification for new mail
│       ├── hooks/
│       │   ├── useInbox.ts           # IPC queries to fetch inbox from cache
│       │   ├── useEmail.ts           # IPC query to fetch full message
│       │   └── useSyncStatus.ts      # Poll sync state from main process
│       └── index.html / main.tsx / index.css
```

**Data flow:**
1. User opens app → main process checks SQLite `sync_state`
2. If no sync yet → show onboarding → OAuth flow in BrowserWindow → store tokens in `bun:sqlite`
3. After auth → full sync: `messages.list` → batch `messages.get(format=FULL)` → insert into SQLite → update `sync_state.historyId`
4. Renderer queries inbox/email via IPC → main process reads from SQLite → returns typed data
5. Background: poll `history.list(startHistoryId=last)` every 30s. On new mail → update SQLite → push event to renderer via IPC
6. App opens cold → SQLite already has data → inbox renders instantly (<100ms)

**Key libraries:**
- Bun built-in `crypto` — PKCE code verifier/challenge (SHA-256), no npm dependency
- Bun built-in `fetch` — Gmail REST API calls (messages.list, messages.get, history.list, batch)
- `bun:sqlite` (built-in) — zero-dependency SQLite driver, synchronous API
- `DOMPurify` (npm) — HTML sanitization for email bodies rendered in WebView (XSS prevention)
- `@tanstack/react-virtual` (npm) — list virtualization for 120fps scrolling
- Electrobun IPC — typed message passing between main and renderer
- macOS `security` CLI — Keychain access for OAuth token storage (zero npm deps, one `Bun.spawn` call)

## Open Questions

1. **Token storage on non-macOS platforms:** macOS uses Keychain via `security` CLI. Linux uses `secret-tool` (libsecret). Windows uses Credential Manager via PowerShell. Default: macOS-only for v0.1 — Keychain via `security add-generic-password`. Other platforms deferred to v0.2.
2. **Gmail API batch requests:** Use Gmail's batch endpoint to reduce API calls during full sync? Default: yes — batch up to 100 `messages.get` calls per request.
3. **Dark mode:** Light mode default with dark toggle, or dark mode default? Default: light mode default, dark mode toggle. Matches "feel at home" for most users.
4. **MIME parsing library:** `mailparser` (npm) requires Node.js streams — may not work in Bun. Alternatives: `letters` (npm, works in Bun), manual parsing with Bun's built-in `Buffer`, or a Bun-native MIME parser. Default: spike `letters` first; fall back to manual header parsing + body extraction from Gmail API's pre-parsed `payload` structure.

## The Assignment

**Ship a working sync pipeline. Target: weekend. Buffer: Monday evening.**

Don't build the React UI first. Don't polish the onboarding. Just:
1. **OAuth spike (2-4 hours):** Implement PKCE manually using Bun's built-in `crypto.subtle` (SHA-256 for code challenge). Register `http://localhost:{port}/oauth/callback` as an authorized redirect URI in Google Cloud Console. At runtime, bind an HTTP server on a random available port using `Bun.serve({ port: 0, fetch(req) { ... } })`. Open a second BrowserWindow for Google's consent screen (`electrobun.config.ts` already supports multiple windows). The consent URL includes `redirect_uri=http://localhost:{port}/oauth/callback`. When the user approves, Google redirects to that localhost URL. The `Bun.serve` handler extracts `code` from `req.url` query params, closes the consent BrowserWindow via Electrobun API, exchanges `code` for tokens via `fetch`, and stores the refresh token in macOS Keychain: `Bun.spawn(["security", "add-generic-password", "-a", "radius", "-s", "gmail-refresh-token", "-w", token, "-U"])`. Read it back on next launch with `security find-generic-password -a radius -s gmail-refresh-token -w`. **Verify before building anything else:** spin up a `Bun.serve` on a random port, confirm you can receive a GET request. This is the highest-risk unknown in the entire stack.
2. **SQLite schema (v0.1 — trimmed):** Create `messages` table only, plus `sync_state` singleton. Drop `threads`, `labels`, `message_labels`, `raw_mime`, `size_estimate` — add in v0.2 migrations. Schema:
   ```sql
   CREATE TABLE messages (
     id TEXT PRIMARY KEY,           -- Gmail message ID
     thread_id TEXT NOT NULL,       -- Gmail thread ID
     history_id TEXT NOT NULL,      -- For incremental sync cursor
     internal_date INTEGER NOT NULL, -- Epoch ms
     from_addr TEXT,                -- Parsed From header
     subject TEXT,                  -- Parsed Subject header
     snippet TEXT,                  -- Gmail-generated snippet
     body_text TEXT,                -- Plain text body (from Gmail API payload)
     body_html TEXT,                -- HTML body (from Gmail API payload)
     created_at INTEGER NOT NULL DEFAULT (unixepoch())
   );
   CREATE INDEX idx_messages_date ON messages(internal_date DESC);
   CREATE TABLE sync_state (
     id INTEGER PRIMARY KEY CHECK (id = 1),
     history_id TEXT,
     last_sync_at INTEGER,
     full_sync_completed_at INTEGER,
     status TEXT NOT NULL DEFAULT 'idle'
   );
   ```
3. **Full sync:** `fetch` to `messages.list` (page through with `pageToken`, `maxResults=500`, `q=in:inbox`). For each page, fetch messages sequentially with concurrency limit of 5 (5 simultaneous `messages.get(format=FULL)` via `Promise.all`). Gmail's batch endpoint is deferred to v0.2 — the multipart/mixed MIME format is non-trivial and not worth the complexity for v0.1. Parse responses, extract `payload.headers` (From, Subject), `payload.body.data` or `payload.parts[].body.data` (base64-decode with Bun's `atob`). Insert into SQLite in batches of 50 with `await Bun.sleep(0)` between batches. After each batch insert, push an IPC `syncStatus` message to the renderer with `progress: { current, total }` where `total` comes from `resultSizeEstimate` in the initial `messages.list` response. Track `historyId` from the newest message. **Sync is non-resumable in v0.1:** if the app quits mid-sync, it restarts from scratch. Accept this for v0.1 — resumable sync (via `sync_page_token` in `sync_state`) is deferred to v0.2.
4. **Non-blocking sync pattern:** `bun:sqlite` is synchronous and blocks the event loop. Strategy: process messages in batches of 50. After each batch insert, `await Bun.sleep(0)` to yield the event loop so IPC and UI remain responsive. After each batch, push an IPC message to the renderer: `{ type: 'syncStatus', status: 'syncing', progress: { current: N, total: T } }`. The renderer uses this for the progress bar. When sync completes, push `{ type: 'syncStatus', status: 'idle' }`.
5. **Incremental sync:** Poll `history.list(startHistoryId=last)` every 30s. Process history entries (messageAdded, labelsAdded, labelsRemoved). Update SQLite. Push new-mail events to renderer via Electrobun IPC.
6. **IPC bridge — typed contracts (never raw SQL):** Define request/response types:
   ```typescript
   // Main process handlers (server-side)
   type IpcRequest =
     | { type: 'getInbox'; limit: number; offset: number }
     | { type: 'getMessage'; id: string }
     | { type: 'getSyncStatus' };
   type IpcResponse =
     | { type: 'inbox'; messages: Message[]; total: number }
     | { type: 'message'; message: Message }
     | { type: 'syncStatus'; status: string; progress?: { current: number; total: number } }
     | { type: 'newMail'; message: Message };  // pushed from main, not requested
   ```
   The main process runs parameterized queries. The renderer never sends SQL. This eliminates SQL injection and keeps the IPC surface small.
7. **Inbox display model — flat messages for v0.2:** The inbox shows individual messages (flat list), not Gmail-style conversation groups. `thread_id` is stored but not used for display grouping. Thread/collapsed-conversation view is deferred to v0.3. This simplifies the IPC query (`getInbox` returns messages sorted by `internal_date DESC`) and the UI (single-line rows per message). If users expect Gmail-style threading, it's called out in the README as a v0.3 feature.
8. **HTML sanitization — main process for security:** DOMPurify runs in the Electrobun main process (Bun), NOT in the renderer (WebView). The body HTML is sanitized before it enters the IPC bridge — unsanitized HTML never transits to the renderer. This creates a process-level security boundary: even if the IPC is compromised, sanitized HTML is the only thing that crosses. Prioritize the DOMPurify-in-Bun compatibility test from the Compatibility Verification section. If Bun-incompatible, use `isomorphic-dompurify` (bundles jsdom) as the primary dependency instead of plain `dompurify`.
9. **MIME parsing spike:** Verify that Gmail API's `payload` structure (with `format=FULL`) gives us clean enough text/html bodies without needing `mailparser`. Many emails will need base64 decoding of `payload.parts[].body.data`. Spike against 50 real emails — multipart, HTML-only, base64, non-ASCII headers. If Gmail's pre-parsed payload is insufficient, add `letters` (npm) as fallback.
10. **HTML sanitization — in main process:** Before sending `body_html` over IPC, sanitize in the main process: `DOMPurify.sanitize(body_html)`. This strips `<script>`, event handlers, tracking pixels (`<img>` with remote src), and CSS that breaks layout. The sanitized HTML is what the renderer receives — unsanitized HTML never crosses the IPC boundary. Non-negotiable for an email client.

If you can open the app and `console.log` your 50 most recent emails from SQLite in the renderer (via the typed IPC bridge), you've proven the stack works. Everything else — the virtualized list, the reader mode, the onboarding animations — is rendering. The sync engine is the foundation.

## Error Handling

The sync engine must handle operational reality. The prior GPUI design doc had a comprehensive error handling section — adapted here for the Electrobun stack:

- **Gmail API rate limits (429):** Retry with exponential backoff (base 1s, max 60s, jitter via `Math.random()`). Track quota usage per 100-second window (Gmail's quota unit is per 100s). Log rate limit events.
- **Token expiration (401):** Automatic refresh using the stored refresh token from Keychain. If refresh fails (token revoked), prompt re-auth via BrowserWindow. Never delete the cached SQLite data — only tokens are invalid.
- **History gaps:** If `history.list` returns a gap (too many changes since `startHistoryId`), or if the stored `historyId` is older than 1 week, trigger a full re-sync of INBOX. Log the event with the expired `historyId`.
- **Network partitions:** Sync pauses gracefully. `sync_state.status` set to `'offline'`. Renderer shows "Offline — showing cached data" rather than crashing. Resume sync when `fetch` succeeds.
- **Partial batch failures:** If a batch `messages.get` request returns partial errors (some IDs failed), retry failed items individually with exponential backoff. Do not fail the entire sync.
- **SQLite durability:** Enable WAL mode (`PRAGMA journal_mode=WAL`) for crash recovery. On startup, validate `historyId` continuity. If mismatch detected, trigger full re-sync.
- **IPC timeout:** If an IPC request to the main process doesn't respond within 5s (e.g., sync is blocking the event loop), the renderer shows a loading state, not a white screen.

## Security & Privacy

This app handles email content — treat it accordingly:

- **Token storage:** OAuth refresh tokens stored in macOS Keychain via `security` CLI (`Bun.spawn`). Never in SQLite. Never in plaintext files. Read on launch, re-prompt OAuth if absent.
- **HTML sanitization:** All `body_html` content passes through `DOMPurify.sanitize()` before rendering in WebView. This strips `<script>`, `onerror`/`onload` handlers, tracking pixels (`<img src>` with remote URLs), and `style` attributes that could break layout. This is non-negotiable — email-borne XSS in a WebView is a real attack vector.
- **Scope minimization:** Request only `https://www.googleapis.com/auth/gmail.readonly` for v0.1. No send, modify, or delete scopes. The OAuth consent screen tells the user exactly what Radius can and cannot do.
- **No telemetry:** No analytics, no crash reporting to third parties. This is a trust feature for a read-only email client. Users need to trust that their email data stays local.
- **SQLite file permissions:** Database file created with `0o600` permissions. Stored in Electrobun's app data directory, not a world-readable location.
